# Harness-First Roadmap Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Turn AI Career Intelligence Suite from a functional local career workflow into a measurable, traceable, recoverable, and safety-bounded career-agent harness.

**Architecture:** Keep the existing Streamlit, CLI, deterministic workflow, and model-neutral LLM adapter. Add harness layers around them in this order: provider capability contract, eval bank, model-harness benchmark matrix, run trace, trace-derived failure corpus, tool catalog v2, runtime enforcement hooks, privacy-preserving memory, evaluator-optimizer, and controlled autonomy. Product features should grow only after these harness layers can measure and constrain quality.

**Tech Stack:** Python 3.12, Streamlit, Typer CLI, Pydantic v2, pytest, ruff, basedpyright, local fake LLM provider, OpenAI-compatible provider boundary.

**Canonical Status:** FINALIZED on 2026-07-10. This Markdown file is the canonical harness roadmap and implementation contract. After this finalization update, future harness work must follow this document and must not modify, supersede, or weaken it. If future work needs clarification, create a separate addendum that references this file; do not edit this file.

## 1. Positioning

This is the **project roadmap**, but its main axis is the **harness roadmap**.

It is not only a harness plan because it changes product behavior, CLI output, Streamlit reporting, memory, and future agent autonomy. It is also not a normal product roadmap because it deliberately does not prioritize more templates, authentication, cloud deployment, job boards, databases, or payment features.

The roadmap should be read as:

> Build the harness first, then use that harness to safely expand the product.

This document is also the long-term harness writing rule for the project. Future harness plans, specs, implementation tasks, and acceptance checks must conform to this roadmap. They may become narrower than this document, but they may not bypass provider capability checks, evals, traces, runtime enforcement, privacy limits, or controlled-autonomy gates.

## 2. Source Synthesis

This roadmap is based on the Anthropic engineering series and the cnblogs Chinese guide that groups the series into five modules:

- Agent architecture: workflows vs agents, prompt chaining, routing, parallelization, orchestrator-workers, evaluator-optimizer.
- Tool capability: tool search, programmatic tool calling, tool-use examples, agent-computer interface design.
- Context and memory: minimal high-signal context, just-in-time loading, structured notes, contextual retrieval.
- Long-running and multi-agent harnesses: feature lists, progress logs, clean handoff, browser/user-level verification, subagent compression.
- Production engineering: evals, sandboxing, coding-agent practices, postmortems, quality monitoring.

The finalized version additionally incorporates four cross-provider and research-backed harness requirements:

- Provider capability contracts: OpenAI-style agent loops, sessions, tracing, guardrails, and approvals differ from direct model/tool-loop APIs; DeepSeek thinking/tool-call modes also have provider-specific state requirements.
- Trace-derived failure corpora: real traces, human/model feedback, and regression evals must form one improvement loop.
- Runtime enforcement: tool execution, memory writes, network fetches, exports, and external boundaries must be intercepted at runtime, not only checked in planner output.
- Model-harness benchmark matrices: agent quality must be reported for each model plus harness configuration, rather than attributed to a base model alone.

## 3. Current Baseline

Existing project strengths:

- Local Streamlit app and `career-ai-agent` CLI already run without a live API key through a fake provider.
- `src/career_ai/agent/` already has planner, executor, execution loop, tool catalog, tool registry, memory summary, and recovery logic.
- `src/career_ai/llm/` already isolates provider-specific model calls behind a conservative adapter.
- `prompts/` already contains externalized prompt strategies.
- Tests already cover workflow, prompt harness, agent runtime, recovery, tool catalog, CLI, history, fetch security, and UI-related behavior.
- Public URL fetching already has hardened SSRF/DNS-rebinding-oriented boundaries through `jd_fetcher.py`, `jd_http_client.py`, and `fetch_models.py`.

Current limitation:

- The system can run the workflow, but it cannot yet produce a strong answer to: "How do we know this run was good, safe, stable, and better than the previous prompt/model version?"

## 4. Strategic Principles

1. Keep the fixed career workflow as the default until evals prove that more model autonomy improves outcomes.
2. Treat prompt changes like code changes: version them, evaluate them, and make regressions visible.
3. Treat context as scarce: pass the smallest high-signal summary needed for each model decision.
4. Treat tools as ACI, not API wrappers: tool names, schemas, examples, and errors must be optimized for model use.
5. Prefer deterministic graders first, model graders second, and human review only for calibration.
6. Do not store full resumes, full job descriptions, or sensitive local paths in memory.
7. Do not add external action capabilities until the action boundary, allowlist, and confirmation rules are explicit.
8. Treat provider behavior as a typed capability contract; never assume OpenAI, DeepSeek, local fake providers, or future OpenAI-compatible providers expose the same tool-call, reasoning, trace, JSON, streaming, or state behavior.
9. Treat failures as harness assets: every meaningful run failure should be reducible to a trace event, a failure-corpus record, and eventually a regression eval.
10. Treat runtime enforcement as mandatory: planner validation is useful, but tool calls, memory writes, network fetches, document exports, and external actions must also pass runtime policy hooks.
11. Report quality at the model-harness configuration level, including pass rate, trace completeness, latency, retry count, and cost proxy where available.

## 5. Target End State

After this roadmap, the project should support this flow:

1. User provides resume and JD through Streamlit or CLI.
2. Agent creates or receives a structured plan.
3. Executor runs local tools under policy.
4. Each tool call emits compact trace events.
5. Prompt strategy and career output are scored against deterministic and optional model-based graders.
6. The run produces a `CareerRunTrace` and `CareerQualityReport`.
7. Memory stores only privacy-preserving user-confirmed summaries.
8. Optional evaluator-optimizer can improve output within a strict iteration cap.
9. Controlled autonomy can choose among safe local tools, but cannot cross external boundaries without policy.
10. Provider capability checks explain which model features are available before a run starts.
11. Failed runs can be converted into regression cases without storing sensitive resume or JD text.
12. Eval reports compare model-harness configurations, not only raw model names.

## 7. Phase Roadmap

### Phase 0: Documentation Anchor

Purpose: Make the roadmap visible and prevent future work from drifting into random feature accumulation.

Deliverables:

- `docs/roadmaps/harness-first-roadmap.md`
- README section: "Harness-first roadmap"

Acceptance criteria:

- `README.md` explains that v0.2-v0.7 focus on eval, trace, tool, context, evaluator, and controlled autonomy.
- The roadmap explicitly says auth, payments, cloud deployment, database-backed multi-user history, and complex RAG are out of scope until harness maturity.
- This implementation plan is marked FINALIZED and canonical; future harness plans must reference it rather than replace it.

### Phase 0.5: Provider Capability Contract

Purpose: Make provider differences explicit before evals, traces, tool calls, or autonomy depend on them.

Deliverables:

- `ProviderCapabilityProfile` model.
- Capability registry for local fake provider, OpenAI-compatible provider boundary, OpenAI direct provider, and DeepSeek-compatible provider.
- Capability flags for structured output, JSON stability, single-turn tool calls, multi-turn tool calls, reasoning/thinking mode, streaming, provider-native tracing, resumable state, and provider-specific message-state requirements.
- Startup/doctor output that reports the active provider capability profile.

Provider-specific rules:

- OpenAI-style SDK orchestration may provide agent-loop, sessions, tracing, guardrails, handoffs, and approval-flow concepts that are not guaranteed in direct model APIs.
- DeepSeek thinking/tool-call modes may require provider-specific reasoning state to be carried across subsequent requests.
- The local fake provider remains the deterministic baseline and must not be removed.

Acceptance criteria:

- Tests validate capability profiles for fake, OpenAI-compatible, and DeepSeek-compatible providers.
- The application never assumes a capability unless the active provider profile declares it.
- Unsupported capabilities degrade to deterministic local behavior or explicit "unsupported" results, not silent fallback.
- No task in this roadmap may introduce provider-specific behavior without updating the capability contract first.

### Phase 1: Eval Bank v0.2

Purpose: Make quality measurable before changing prompts or autonomy.

Deliverables:

- Eval case schema.
- 5 initial golden cases.
- Deterministic graders.
- CLI or pytest entrypoint to run evals.

Minimum eval dimensions:

- JD role title extraction.
- Match score range stability.
- Missing keyword detection.
- No fabricated resume facts in rewritten bullets.
- Prompt strategy comparison returns at least three strategies.
- CLI and workflow behavior remain consistent.

Acceptance criteria:

- `python -m pytest tests/test_eval_loader.py tests/test_eval_graders.py tests/test_eval_runner.py`
- Existing full suite still passes.
- Eval results include per-case pass/fail details, not only a single score.

### Phase 1.5: Model-Harness Benchmark Matrix

Purpose: Report agent quality at the provider/model/harness-configuration level instead of attributing outcomes to a model name alone.

Deliverables:

- Matrix runner over shared eval cases.
- Harness configuration model that captures provider, model, prompt set, tool catalog version, policy version, trace version, retry budget, and optimizer settings.
- Aggregate report with pass rate, failed checks, trace completeness, retry count, latency proxy, token/cost proxy when available, and unsupported capability notes.

Required matrix rows:

- Local fake provider with deterministic harness.
- OpenAI-compatible provider boundary when configured.
- DeepSeek-compatible provider boundary when configured.

Acceptance criteria:

- Matrix runner can execute with only the fake provider.
- Provider rows with missing credentials are reported as skipped, not failed.
- Results distinguish model failures, provider capability gaps, and harness policy denials.
- Regression claims must cite the exact model-harness configuration that produced them.

### Phase 2: Trace Harness v0.3

Purpose: Make every run inspectable and debuggable.

Deliverables:

- `CareerRunTrace` model.
- Tool-call trace events.
- Prompt strategy score trace.
- Recovery trace.
- Compact CLI trace output.
- Streamlit trust panel.

Trace must include:

- Run ID.
- Timestamp.
- Provider and model.
- Agent mode.
- Planned steps.
- Executed tool calls.
- Tool status and retry count.
- Prompt strategy summary.
- Quality report summary.
- Privacy-preserving input summary, not full resume/JD.

Acceptance criteria:

- Trace JSON can be serialized and validated with Pydantic.
- Trace never stores full resume text, full JD text, uploaded local file path, or API key.
- CLI can print a compact trace summary.

### Phase 2.5: Trace-Derived Failure Corpus

Purpose: Turn real failures and review feedback into durable regression assets without storing sensitive raw inputs.

Deliverables:

- Failure corpus schema.
- Sanitizer that extracts trace event summaries, failure category, provider capability profile, harness configuration, expected behavior, and redacted minimal reproduction notes.
- CLI command or developer function to propose a regression eval from a failed trace.
- Review state: `candidate`, `accepted`, `rejected`, `converted_to_eval`.

Allowed corpus content:

- Run ID.
- Provider/model/harness identifiers.
- Failed check names.
- Tool names and failure categories.
- Redacted input summary.
- Human or model feedback summary.
- Expected behavior and acceptance criteria.

Forbidden corpus content:

- Full resume text.
- Full job description text.
- Full cover letter.
- Local file paths.
- API keys, credentials, cookies, tokens.
- Contact information unless explicitly marked safe and necessary.

Acceptance criteria:

- A failed run can produce a sanitized regression candidate.
- A regression candidate can be converted into an eval case only after review.
- Tests prove sensitive strings are removed from the failure corpus.
- The failure corpus links traces to evals so future prompt/model changes can prove they did not reopen old failures.

### Phase 3: Tool Catalog v2

Purpose: Improve the model-facing tool interface before expanding autonomy.

Deliverables:

- Namespaced tool names or display aliases.
- Input examples for each tool.
- Response modes: `compact` and `detailed`.
- Standard failure categories.
- Tool safety annotations.

Suggested tool naming:

- `career.analyze_fit`
- `career.compare_prompt_strategies`
- `document.export_resume_docx`
- `document.export_cover_letter_docx`
- `memory.summarize_private`
- `jd.fetch_public`
- `resume.extract_local`

Failure taxonomy:

- `invalid_input`
- `missing_input`
- `temporary_failure`
- `unsafe_target`
- `unsupported_format`
- `privacy_boundary`
- `quality_gate_failed`

Acceptance criteria:

- The rendered catalog remains compact.
- Every tool has at least one realistic input example.
- Existing planner tests are updated to assert safety and example visibility.

### Phase 3.5: Runtime Enforcement Hooks

Purpose: Enforce policy at execution time so unsafe behavior is blocked even if the planner or model proposes it.

Deliverables:

- Runtime enforcement engine.
- Hook points for pre-tool-call, post-tool-call, memory-write, network-fetch, document-export, and external-action requests.
- Standard enforcement outcomes: `allowed`, `denied`, `redacted`, `requires_confirmation`, `unsupported`.
- Enforcement trace events that include policy version, boundary type, decision, and compact reason.

Default enforcement rules:

- Public JD fetches must reuse the existing SSRF/DNS-rebinding-safe fetch boundary.
- Memory writes must pass privacy redaction and allowed-field checks.
- Document exports may only use generated artifacts from the current run or explicitly safe local paths.
- External actions such as applying for jobs, sending email, modifying credentials, changing git state, or running shell commands from the agent are denied.
- Denied actions should recover to a safe fallback when possible.

Acceptance criteria:

- Tests prove unsafe tool calls, unsafe memory writes, unsafe network targets, and disallowed external actions are denied at runtime.
- Denials create trace events and do not crash the app.
- Runtime enforcement remains active even when model-planned tool validation is bypassed in tests.

### Phase 4: Privacy-Preserving Context Memory v0.5

Purpose: Support long-term career context without storing sensitive raw inputs.

Deliverables:

- `CareerProfileMemory` model.
- Memory summarizer that extracts only user-confirmed or output-derived safe summaries.
- Memory read path for future runs.
- Memory redaction tests.

Allowed memory:

- Target role families.
- Confirmed skills.
- Repeated missing keywords.
- Preferred tone or language.
- Application asset preferences.
- Past match score ranges.

Forbidden memory:

- Full resume text.
- Full job description text.
- Full cover letter.
- Local file paths.
- Email addresses or phone numbers unless explicitly marked safe.
- API keys, credentials, cookies, tokens.

Acceptance criteria:

- Tests prove sensitive strings are not persisted.
- Existing `.career_ai/history.json` remains local and ignored.
- Memory can be used to improve a future prompt without increasing context bloat.

### Phase 5: Evaluator-Optimizer v0.6

Purpose: Add bounded output improvement where clear criteria exist.

Deliverables:

- `CareerQualityReport`.
- Deterministic verifier.
- Optional model-based evaluator behind provider boundary.
- Optimizer loop capped at 1-2 iterations.

Quality dimensions:

- Factual consistency.
- JD alignment.
- Missing skill clarity.
- Rewrite usefulness.
- Cover letter specificity.
- No unsupported claims.
- Output structure completeness.

Acceptance criteria:

- Evaluation can fail a run without crashing the app.
- Failed quality checks produce actionable messages.
- Optimizer never loops indefinitely.
- Deterministic fake provider remains usable with no API key.

### Phase 6: Controlled Autonomy v0.7

Purpose: Let the model make bounded workflow decisions only after eval and trace are strong.

Deliverables:

- Policy object for allowed tool choices.
- Planner output validation against tool catalog.
- Tool order constraints.
- Deny-and-continue behavior for unsafe or unsupported actions.

Allowed autonomy:

- Choose whether JD comes from URL, manual text, or sample input.
- Choose whether prompt comparison is necessary.
- Choose whether to export documents after analysis.
- Choose whether to save privacy-preserving memory summary.

Disallowed autonomy:

- Reading arbitrary local files.
- Sending data to external services.
- Fetching private or unsafe network targets.
- Applying for jobs automatically.
- Modifying shell, git, environment, or credential state.

Acceptance criteria:

- Invalid planned tools are rejected.
- Unsafe planned actions create trace events.
- The agent recovers to a safe fallback when possible.

### Phase 7: Product Expansion After Harness Maturity

Purpose: Add user-facing features only when quality is measurable.

Candidate features:

- Batch JD comparison.
- Application packet manager.
- Interview prep from JD and resume.
- Long-term career gap dashboard.
- Job-search history insights.
- Optional contextual retrieval over user-approved documents.

Gate:

- Do not start these until eval, trace, tool catalog v2, and privacy memory are stable.

## 10. Non-Goals

Do not include in this roadmap:

- Authentication.
- Payments.
- Cloud deployment.
- Multi-user database history.
- Complex RAG over private documents.
- Automated job application submission.
- Email, Google Drive, Notion, or calendar integrations.
- Multi-agent research system for job boards.

These can be revisited only after eval, trace, tool catalog v2, privacy memory, and controlled autonomy are stable.

Do not add provider-specific shortcuts that bypass the capability contract. Do not add model-specific prompt or tool behavior unless it is represented in the provider capability profile, eval matrix, and trace output.

## 11. Success Metrics

Harness maturity:

- Provider capability profiles exist for fake, OpenAI-compatible, and DeepSeek-compatible providers.
- 20-50 eval cases exist.
- Regression evals pass near 100%.
- Capability evals include known hard cases.
- Every run has trace and quality report.
- Failed traces can become sanitized failure-corpus candidates.
- Accepted failure-corpus records can become regression evals.
- Prompt changes can be compared against previous results.
- Model-harness matrix results report pass rate, skipped rows, unsupported capabilities, retry count, trace completeness, and latency/cost proxy when available.

Product maturity:

- User can understand why a result is trustworthy.
- User can see what the agent did and what it skipped.
- User can run the app locally without an API key.
- User can opt into a live model provider without changing workflow contracts.

Safety maturity:

- Full resume/JD not persisted in memory.
- Unsafe JD URLs are rejected.
- External action tools are absent or gated.
- Model-planned tools are validated before execution.
- Runtime enforcement denies unsafe tool calls, memory writes, network fetches, document exports, and external-action requests.
- Enforcement denials are recorded in trace events and recover safely when possible.

Governance maturity:

- This finalized roadmap remains the canonical harness contract.
- Future harness plans reference this file and do not modify, replace, or weaken it.
- Any future addendum is narrower than this roadmap and preserves all gates defined here.
